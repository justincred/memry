Hint Mode — Implementation Approaches

Three architectural strategies for the overlay system

A

Portal Overlay (React-native)

A single React portal renders all hint badges as absolutely-positioned <span>s in a top-level overlay <div>. A useHintMode() hook manages state, scans the DOM for clickable elements via querySelectorAll, computes positions with getBoundingClientRect, and assigns labels.

Pros

  • Pure React — fits existing architecture
  • Badges are React components (easy to style/animate)
  • Single render layer, no z-index wars

Cons

  • DOM scan + position calc on every activation
  • Scroll/resize can desync badge positions
  • Must handle portal click-through correctly

Recommended — cleanest integration with existing codebase

B

Inline Injection

Instead of an overlay, inject a small <span class="hint-badge"> directly next to each clickable element in the DOM. Each interactive component gets a wrapper or ref that conditionally renders its badge when hint mode is active.

Pros

  • Badges naturally follow layout changes
  • No position calculation needed
  • Scroll-proof — badges move with their elements

Cons

  • Invasive — touches many components across the app
  • Layout shifts when badges appear
  • Hard to add to third-party components (BlockNote toolbar)
C

Hybrid: Portal + MutationObserver

Portal overlay like A, but with a MutationObserver watching for DOM changes and a ResizeObserver for layout shifts. Badges auto-reposition when the DOM mutates. More robust but more complex.

Pros

  • Handles dynamic content (dropdowns, expanding trees)
  • Self-healing positions
  • Works with any component, no modifications needed

Cons

  • Observer overhead on complex DOM
  • Over-engineered for a modal that lasts <2 seconds
  • Edge cases with observer lifecycle
