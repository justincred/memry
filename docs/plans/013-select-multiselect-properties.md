# Design: Select & Multiselect Note Properties

Generated by /office-hours on 2026-03-27
Branch: feat/note-page
Repo: memrynote/memry
Status: APPROVED
Mode: Startup

## Problem Statement

Power note-takers who use memry as a personal database (books, movies, research, recipes) need structured status/category tracking on notes. The canonical `PropertyTypes` in `packages/contracts` currently exports text/number/checkbox/date/url. The preload declarations already reference `select`, `multiselect`, and `rating` types, but nothing renders or edits them — the gap is UI + backend wiring, not type definitions. Users resort to a mix of tags, text properties, and in-body text to approximate what should be a first-class select property.

This is table-stakes for competing with Notion's property system, and notably a gap that Obsidian still hasn't solved natively (users need 2-3 plugins: Metadata Menu + Dataview + Dropdown Vars).

## Demand Evidence

- **Competitive necessity**: Notion, Capacities, and Anytype all ship select/multiselect. Obsidian lacks it natively — power users complain on forums and rely on plugin stacks.
- **Current workarounds**: Users use a mix of tags-as-status, plain text properties, and in-body text — diffuse but real pain.
- **Template gap**: memry's template system already defines `select` and `multiselect` types in `templates-api.ts` with `options: string[]`, but they're not wired into the canonical property system. The infrastructure is half-built.

## Status Quo

Without select properties, a memry user tracking book status would:
1. Add a text property called "Status" and type "reading" — no validation, no color, no dropdown
2. Use tags like `#reading`, `#read` — pollutes the tag namespace, no mutual exclusivity
3. Write it in the note body — invisible to any structured view or query
4. Some combination of all three — inconsistent across notes

## Target User & Narrowest Wedge

**Target**: Power note-takers who use notes as personal databases. People who organize collections (books, movies, recipes, contacts, research papers) and want structured metadata — not just free-text.

**Narrowest wedge**: A single `select` property type with user-defined options and colors. This alone covers the "reading status" / "movie status" / "priority" use case. Multiselect (for genres, categories) is the natural second step but not required for initial value.

## Constraints

- **Markdown is the source of truth**: All property data must be derivable from markdown files. The DB is a rebuildable cache.
- **Pre-production**: No backward-compat needed. DB schema and code can change freely.
- **Existing infrastructure**: `propertyDefinitions` table has an `options` column. Templates already define select/multiselect. The Picker component handles dropdowns.
- **Sync**: `.memry/` folder syncs across devices via existing CRDT infrastructure. Property definitions stored here will sync automatically.

## Premises

1. **Three new property types**: `status`, `select`, and `multiselect`. Status is special (predefined semantic categories like task statuses); select/multiselect are generic.
2. **Status property type**: Ships with 3 category groups (To-do / In progress / Complete), each with default options (Not started, In Progress, Done). Users can add/rename/delete options within each category. Independent from task statuses — same UX pattern, separate data.
3. Option DEFINITIONS (available choices, colors, categories) live in `.memry/properties.md` as YAML frontmatter — the vault-level source of truth. The `propertyDefinitions` table becomes a rebuildable cache of this file.
4. Option VALUES continue to live in each note's frontmatter (e.g., `status: Reading`).
5. UI uses colored chips in a Picker dropdown, consistent with existing TagChip patterns. Select chips and tags share the same visual style — distinguished by position (tags in TagsRow, select chips in PropertyRow).
6. Template unification (`TemplatePropertyType` → shared `PropertyType`) is **deferred** to a follow-up.

## Approaches Considered

### Approach A: Inferred Options (Minimal Viable)
Options auto-discovered by scanning frontmatter values across all notes sharing a property name. No definitions file.
- **Effort**: S (human: ~3 days / CC: ~30 min)
- **Risk**: Low
- **Pros**: Ships fastest, self-organizing, no new files
- **Cons**: No pre-definition or colors, typos create junk options, no validation
- **Completeness**: 4/10

### Approach B: Vault-Level Definitions File (Chosen)
`.memry/properties.md` with YAML frontmatter defines property schemas. `propertyDefinitions` table becomes a rebuildable cache. Full settings UI.
- **Effort**: M (human: ~1 week / CC: ~45 min)
- **Risk**: Low-Med
- **Pros**: True markdown-as-source-of-truth, pre-defined colored options, typo prevention, syncs across devices, portable
- **Cons**: New file format to design, parser/watcher needed, settings surface area
- **Completeness**: 9/10

### Approach C: Template-Driven Inheritance
Select options defined in templates. Notes inherit from their creation template.
- **Effort**: M (human: ~1 week / CC: ~40 min)
- **Risk**: Med
- **Pros**: Leverages existing template system, natural mental model
- **Cons**: Orphan problem (notes not from templates), template-coupling
- **Completeness**: 6/10

## Recommended Approach

**Approach B: Vault-Level Definitions File**

Chosen because it fully respects "markdown is source of truth," provides the best UX (pre-defined colored options, validation, typo prevention), and syncs across devices via existing `.memry/` infrastructure. The effort delta over Approach A is negligible with CC.

### Architecture Overview

```
.memry/properties.md          <-- Source of truth (YAML frontmatter)
         |
         v
propertyDefinitions table     <-- Rebuildable cache (SQLite)
         |
         v
PropertyRow / SelectEditor    <-- UI components
         |
         v
note frontmatter              <-- Per-note values (e.g., status: Reading)
```

### Source of Truth Format

`.memry/properties.md` uses YAML frontmatter to define property schemas:

```yaml
---
properties:
  status:
    type: status
    categories:
      todo:
        label: To-do
        options:
          - { value: Not started, color: stone, default: true }
          - { value: Want to Read, color: slate }
      in_progress:
        label: In progress
        options:
          - { value: In Progress, color: amber }
          - { value: Reading, color: amber }
      done:
        label: Complete
        options:
          - { value: Done, color: emerald }
          - { value: Read, color: emerald }
          - { value: Abandoned, color: rose }
  genre:
    type: multiselect
    options:
      - { value: Design, color: violet }
      - { value: Psychology, color: sky }
      - { value: Science, color: emerald }
  priority:
    type: select
    options:
      - { value: High, color: rose }
      - { value: Medium, color: amber }
      - { value: Low, color: stone }
---
```

**Key differences between the 3 types:**
- **`status`**: Options grouped into 3 semantic categories (todo/in_progress/done). Ships with defaults. Category headers shown in picker. Follows task status UX.
- **`select`**: Flat list of options with colors. No categories. Generic.
- **`multiselect`**: Same as select but allows multiple values. Stored as YAML array in note frontmatter.

**Colors use named palette** from `tag-colors.ts` (rose, amber, emerald, etc.) — not raw hex. This ensures theme consistency across warm/white/dark modes.

The body of the file is empty or can contain documentation about the property schema. Only the frontmatter is parsed.

### Per-Note Frontmatter

Individual notes store the selected value(s) in their own frontmatter:

```yaml
---
id: abc123
title: The Design of Everyday Things
tags: [books]
status: Reading
genre: [Design, Psychology]
---
```

- `select` values are stored as plain strings
- `multiselect` values are stored as YAML arrays

### Implementation Layers

**Layer 1: Contracts (`packages/contracts`)**
- Export `STATUS`, `SELECT`, and `MULTISELECT` from the canonical `PropertyTypes` constant (preload already references select/multiselect — make `packages/contracts` the single export point and remove preload-local definitions)
- Add `SelectOption` type: `{ value: string, color: string }` — color is a named color from tag-colors.ts palette, not raw hex
- Add `StatusCategory` type: `{ label: string, options: SelectOption[] }` with category keys: `todo | in_progress | done`
- Add `PropertyDefinition` type to contracts (name, type, options or categories, defaultValue)
- Add `PropertyDefinitionsFile` schema (Zod) for parsing `.memry/properties.md` — supports both flat options (select/multiselect) and categorized options (status)
- Add `DEFAULT_STATUS_DEFINITION` constant — ships with Not started / In Progress / Done defaults
- Template unification deferred — `TemplatePropertyType` stays separate for now

**Layer 2: Backend (`apps/desktop/src/main`)**
- **PropertyDefinitionsService**: Singleton service that owns the `.memry/properties.md` lifecycle. Parses YAML frontmatter, writes definitions, rebuilds DB cache. All writes serialized via a promise queue (prevents multi-window race conditions).
- **Event-driven reload** (NOT file watcher — VaultWatcher ignores `.memry/` because it filters hidden dirs): Re-read `.memry/properties.md` on: (1) vault open, (2) after own writes (immediate, no watcher needed), (3) after sync engine batch completes (hook into existing post-pull event in engine, NOT a fictional `sync-pull-complete` event).
- **Cache rebuild**: Parse definitions file → upsert all into `propertyDefinitions` table → remove stale entries not in file. Lazy init: if `.memry/properties.md` doesn't exist, create it with empty `properties: {}` frontmatter on first select property creation.
- **Validation**: Allow unknown values (show warning chip) but don't reject — prevents data loss if definitions change.
- **IPC handlers**: Same channel names (`notes:get/create/update-property-definition`), new file-first implementation. Handlers become thin delegates to PropertyDefinitionsService.
- **Write path**: IPC handler → PropertyDefinitionsService.write() → atomicWrite .memry/properties.md → immediate re-read → update DB cache → notify renderer. No file watcher involved.
- **Deserialization fix**: Add `select` and `multiselect` cases to `deserializeValue()` in `notes.ts` — `multiselect` must parse JSON array, not return raw string.
- **Rename cascade**: Use `isWritebackIgnored()` pattern to prevent VaultWatcher storm when batch-updating note frontmatter. Synchronous with progress bar if >50 notes.

**Layer 3: DB Schema (`packages/db-schema`)**
- `propertyDefinitions` table already has `options` column — ensure it stores `SelectOption[]` as JSON
- `noteProperties` table unchanged — values are already stored as JSON text
- No new tables needed

**Layer 4: Renderer (`apps/desktop/src/renderer`)**
- **StatusEditor component**: Picker dropdown with category group headers (To-do / In progress / Complete). Options listed under each category with color dots. Follows Notion's Status property UX. Reuses existing Picker component.
- **SelectEditor component**: Picker dropdown with flat option list (no categories). Color dots + search.
- **MultiselectEditor component**: Same as SelectEditor but allows multiple selections. Checkmarks on selected items. Multiple colored chips displayed in PropertyRow.
- **SelectChip component**: Reuse TagChip's visual pattern (colored pill with `withAlpha(color, 0.12)` background, 11px font, 10px border-radius). Use named colors from `tag-colors.ts` palette.
- **PropertyRow extension**: Add `status`/`select`/`multiselect` cases to value display and editor
- **AddPropertyPopup extension**: Include Status, Select, and Multiselect in the type selector with appropriate icons
- **Settings page**: Property definitions manager following TagManager pattern — list, search, expand/collapse, edit options with ColorPicker, reorder, delete with confirmation
- **Inline option creation**: "+" button in picker to add new option. Auto-assigns color from palette rotation (same 20-color palette as tags)

**Layer 5: Sync**
- **VaultWatcher does NOT cover `.memry/`** — confirmed. Hidden-file filter (line 213) + hardcoded watchPaths (notes/, journal/ only).
- **Sync approach**: Hook into the sync engine's post-pull batch event to trigger a re-read of `.memry/properties.md`. The note sync handler writes properties to disk during pull — after that write completes, PropertyDefinitionsService re-reads and rebuilds cache.
- **Alternative considered**: `tagDefinitionHandler` sync pattern (dedicated SyncItemHandler for property definitions). Rejected because it makes DB the sync source of truth, which contradicts the "markdown is source of truth" constraint. File-first approach preserved.
- **Implementation task**: Find the real post-batch event in `engine.ts` (not the fictional `sync-pull-complete`). The engine already has per-batch processing — hook PropertyDefinitionsService.reload() into the post-batch callback.
- **Conflict resolution**: CRDT operates at the Yjs text level on the file. Parse-on-change handles malformed YAML gracefully — log warning, keep last-known-good cache. Known limitation for concurrent YAML edits; manual resolution may be needed in rare edge cases.

### UI Interactions

**Viewing a property value:**
- `status` / `select`: Single colored chip (TagChip pattern) in the PropertyRow value slot
- `multiselect`: Row of colored chips with 4px gap, wraps to next line if needed
- No value set: Gray "Empty" text (13px, text-tertiary) — same as existing text property empty state

**Editing a status property:**
- Click the chip (or empty "Empty" text) to open Picker dropdown
- Picker shows options grouped by category with headers: **To-do** / **In progress** / **Complete**
- Each category header has a "+" button to add options within that category
- Options show colored dot + name. Current value has checkmark.
- Clicking an option replaces the current value + closes picker
- First-time UX: Status properties ship with defaults — picker always has options on first click. No empty state.

**Editing a select/multiselect property:**
- Click the chip (or empty slot) to open Picker dropdown
- Picker shows flat list of options with color dots and search
- For select: clicking an option replaces the current value
- For multiselect: clicking toggles selection (checkmark shows selected state), picker stays open
- Divider below options, then "+" New option and "Edit options" actions
- Keyboard: Picker already handles arrow-key navigation, Enter to confirm, Escape to close, and type-ahead search natively

**First-time UX for select/multiselect (no options defined):**
- Picker opens with "No options yet" message + auto-focused inline text input with color dot
- User types option name → press Enter → option created + selected. Picker stays open for more.
- This IS the onboarding — no separate settings trip needed

**Orphan values (value not in definitions):**
- If a note has a property value that doesn't match any defined option (e.g., after a rename/delete), show it as a gray "unknown" chip with a subtle warning indicator
- Clicking the unknown chip opens the picker with an "Add to definitions" action at the top
- This prevents data loss when definitions change — the note keeps its value until the user resolves it

**Rename/delete cascade:**
- **Rename**: When a user renames an option in settings (e.g., "Reading" → "In Progress"), batch-update all note frontmatter files that use the old value. Show a confirmation: "This will update X notes."
- **Delete**: When deleting an option, warn: "X notes still use this value. They will show as unknown." Do NOT auto-remove values from notes — let the user clean up.

**Creating a new option inline:**
- Click "+ New option" in the picker
- Inline text input appears at the bottom of the option list
- Type name, pick color from palette, press Enter
- New option is added to `.memry/properties.md` definitions and immediately available
- This is the fast path — settings page is the full management path

**Managing definitions in Settings:**
- New "Properties" section in settings (alongside Tags)
- List of all property definitions with their type and option preview
- Click to expand: edit options, reorder, change colors, delete
- "Add property definition" button at the bottom

### Visual Specifications

- **Chip**: Same as TagChip — `withAlpha(color, 0.12)` bg, color text, 11px/14px font, `px-2 py-0.5`, 10px border-radius
- **Chip distinction from tags**: Position only. Tags in TagsRow, property chips in PropertyRow.
- **Picker**: Reuse existing Picker component. No custom dropdown.
- **Status category headers**: 11px uppercase text-tertiary with `+` button right-aligned. Same visual weight as section dividers in existing UI.
- **Color dots in picker**: 8px circle with the named color. Same as task status dots.
- **Named color palette**: Use `tag-colors.ts` names (rose, amber, emerald, etc.) — not raw hex. Ensures theme consistency.
- **Auto-color assignment**: Rotate through palette in order (rose → coral → tangerine → ...) for new options. User can override via ColorPicker.

### Accessibility

- Picker keyboard: arrow keys, Enter, Escape, type-ahead search — all handled by existing Picker component
- Chip role: `role="option"` with `aria-selected` (already in TagChip)
- Status property: `role="listbox"` with `role="group"` for each category, `aria-label` on group headers
- Color contrast: all 20 named colors tested against dark/light backgrounds in `withAlpha(color, 0.12)` — passes WCAG AA for text
- Touch targets: chips min 28px height (existing TagChip sizing), picker items min 32px height

## Decisions (closed)

- **Orphan values**: Gray "unknown" chip with "Add to definitions" action. See UI Interactions section.
- **Template unification**: Deferred to follow-up. See Premise 5.
- **Folder-scoped definitions**: Out of scope for V1. Vault-wide only. Folder-scoping would require a different file format and is not needed to ship the core feature.
- **File scope**: `.memry/properties.md` only defines select/multiselect property options. Text/number/checkbox/date/url properties don't need definitions — they have no constrained option sets. If future types need definitions, they can be added to this file.

## Open Questions

1. **Should property definitions also support `defaultValue`?** E.g., new book notes auto-get "Want to Read" status. Templates already have this — could add a `default` field to each property definition.
2. **Color auto-assignment**: When creating a new option inline, auto-assign a color from a palette rotation? Or require manual pick? Recommendation: auto-assign from a rotation of the 10 predefined colors (same palette as task statuses), let user override.
